# Commit 提交规范（美化版）
遵循统一的 Commit 提交规范，可标准化提交信息、自动化生成变更日志、简化版本管理，提升团队协作效率。

---

## ✨ feat 最实用的延伸价值
> 通过工具（如 `standard-version`、`semantic-release`），扫描所有含 `feat` 的提交，**自动生成结构化变更日志**，无需手动整理版本更新内容，大幅提升发布效率。

示例自动生成的 Changelog 片段：
```markdown
# Changelog
## [1.2.0] - 2025-11-18
### Features
- 支持微信登录（对应 feat 提交）
- 新增数据导出 Excel 功能（对应 feat 提交）
```

---

## 🔗 关联语义化版本（Semantic Versioning）
语义化版本格式：`主版本号.次版本号.修订号`（如 `v1.2.3`），与提交类型直接挂钩：

| 版本层级       | 格式示例   | 对应提交类型及说明                                                                 |
|----------------|------------|----------------------------------------------------------------------------------|
| 主版本号（Major） | 1.x.x      | 不兼容的 API 变更，需配合 `feat!` 或 `BREAKING CHANGE` 标记（如重构核心逻辑、删除旧接口） |
| 次版本号（Minor） | x.2.x      | 向后兼容的功能新增，对应 `feat` 提交（如新增登录方式、扩展接口参数）                 |
| 修订号（Patch）  | x.x.3      | 向后兼容的问题修复，对应 `fix` 提交（如修复登录失败、数据展示异常等 Bug）             |

## 📋 常用提交类型说明
| 提交类型   | 核心功能描述                          | 实用示例                                  |
|------------|---------------------------------------|-------------------------------------------|
| `feat`     | 新增功能（向后兼容）                  | `feat: 支持微信扫码登录`                  |
| `fix`      | 修复 Bug（向后兼容）                  | `fix: 修复手机号验证码登录失败的问题`      |
| `docs`     | 仅修改文档（如 README、API 文档）     | `docs: 更新用户注册接口参数说明`          |
| `refactor` | 重构代码（不新增功能、不修复 Bug）    | `refactor: 优化用户模块代码结构，提升可读性`|
| `style`    | 代码格式调整（不影响逻辑）            | `style: 格式化代码（ESLint 修复）`        |
| `test`     | 新增/修改测试用例（单元测试、集成测试）| `test: 补充登录功能单元测试，覆盖率达90%`  |
| `chore`    | 构建流程/依赖/工具配置修改            | `chore: 升级 npm 依赖到 v10.2.0`          |

## 📌 补充说明
1. **破坏性变更标记**：若提交涉及不兼容变更（如 `feat!`），需在提交信息中明确标注 `BREAKING CHANGE`，示例：
   ```bash
   git commit -m "feat!: 重构用户认证逻辑
   
   BREAKING CHANGE: 旧版 /api/login 接口废弃，需改用 /api/v2/login"
   ```
2. **工具支持**：配合 `commitlint`（校验提交规范）、`commitizen`（交互式提交），可进一步降低规范落地成本。
3. **作用域补充**：提交类型后可添加作用域（如 `feat(支付模块): 支持支付宝支付`），让提交意图更精准。
